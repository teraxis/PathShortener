# Фінальне технічне завдання: нормалізація скорочень, аналіз шляхів, ШІ-моделі, теми та локалізація

## 1. Загальна мета

Потрібно доопрацювати програму так, щоб вона коректно аналізувала назви файлів, папок і повні шляхи, пропонувала безпечні скорочення лише тоді, коли це справді потрібно, і не створювала незрозумілі або некоректні абревіатури.

Програма має підтримувати два режими аналізу:

1. Звичайний аналіз — без використання ШІ, на основі алгоритмів і словника.
2. Інтелектуальний аналіз — із використанням моделей Ollama, Claude, OpenAI, Grok або Gemini.

Також потрібно оновити інтерфейс Avalonia, додати вибір теми, зробити програму двомовною та винести всі користувацькі тексти в ресурси локалізації.

## 2. Основна проблема

Зараз система помилково перетворює багато назв на абревіатури. Частина таких скорочень є незрозумілою, штучною і неприйнятною.

Приклад прийнятних скорочень:

```json
{
  "конституційни суд україн": {
    "original": "Конституційний Суд України",
    "short": "КСУ",
    "source": "dict"
  },
  "верховни суд": {
    "original": "Верховний Суд",
    "short": "ВС",
    "source": "dict"
  }
}
```

Такі скорочення допустимі, оскільки вони є сталими, загальноприйнятими або явно заданими у словнику.

Приклад неприйнятних скорочень:

```json
{
  "левчук о о кандзюб": {
    "original": "_ЛЕВЧУК_О_О_Кандзюба",
    "short": "_О_О_К",
    "source": "llm"
  },
  "леміщенк о о бурлаков": {
    "original": "_ЛЕМІЩЕНКО_О_О_Бурлаков",
    "short": "_ЛЕБ_Бурлаков",
    "source": "llm"
  }
}
```

Такі скорочення потрібно заборонити, оскільки вони штучні, незрозумілі користувачу і можуть пошкодити зміст назви.

## 3. Основні правила скорочення

### 3.1. Не скорочувати без потреби

Якщо довжина назви файлу, назви папки або повного шляху не перевищує встановлені допустимі межі, скорочення виконувати не потрібно.

У такому випадку програма не повинна:

* запускати алгоритмічне скорочення;
* передавати назву або шлях до ШІ-моделі;
* створювати нові абревіатури;
* змінювати оригінальну назву;
* додавати елемент у список для інтелектуального аналізу.

Допустимі межі довжини мають бути налаштовуваними.

Потрібно передбачити окремі ліміти для:

* назви файлу;
* назви папки;
* повного шляху;
* внутрішнього шляху в архіві, якщо архіви обробляються окремо.

### 3.2. Дозволені скорочення

Скорочення дозволене лише в таких випадках:

1. Скорочення знайдене у внутрішньому словнику.
2. Скорочення є сталою загальноприйнятою абревіатурою.
3. Скорочення створене звичайним алгоритмом за правилами, які явно ввімкнені користувачем.
4. Скорочення запропоноване ШІ-моделлю, але воно не є довільною абревіатурою з імен, прізвищ, ініціалів або випадкових слів.
5. Скорочення не спотворює зміст назви.
6. Скорочення фактично зменшує довжину і допомагає вкластися в допустимі межі.

### 3.3. Заборонені скорочення

Заборонено:

* створювати абревіатури з прізвищ, імен та ініціалів;
* скорочувати назви сторін або учасників справи до незрозумілих наборів літер;
* скорочувати кожне слово до першої літери без явного правила;
* генерувати штучні абревіатури через LLM без підтвердження їх прийнятності;
* скорочувати текст, якщо довжина не перевищує допустиму межу;
* передавати моделі параметр на скорочення для назв, які не потребують скорочення;
* замінювати змістовні фрагменти назв випадковими або непрозорими скороченнями;
* автоматично додавати LLM-скорочення до постійного словника без підтвердження користувача.

## 4. Інтерфейс Avalonia

### 4.1. Загальний стиль

Потрібно оновити інтерфейс Avalonia так, щоб він був сучасним, чистим і зручним для тривалої роботи з великою кількістю файлів.

Основні вимоги:

* акуратні відступи;
* чітка типографіка;
* зрозумілі кнопки;
* читабельна таблиця;
* логічне групування налаштувань;
* візуальне розділення блоків;
* стабільна робота при великій кількості елементів;
* коректне масштабування вікна;
* зручна робота зі списками папок, архівів і профілів моделей.

### 4.2. Вибір теми

У програмі потрібно реалізувати можливість вибору теми інтерфейсу.

У вкладці “Налаштування” потрібно додати блок “Інтерфейс”.

У цьому блоці користувач має мати можливість вибрати тему:

* світла тема;
* темна тема;
* системна тема, якщо це можливо реалізувати в Avalonia.

Світла тема має бути темою за замовчуванням.

Темна тема не повинна бути повністю чорною. Потрібно використати м’яку темну палітру, зручну для тривалої роботи.

Вимоги до темної теми:

* фон не має бути чисто чорним;
* основний фон має бути темно-сірим або графітовим;
* панелі, картки та таблиці мають візуально відрізнятися від основного фону;
* текст має бути контрастним, але не надто різким;
* акцентні кольори мають бути стриманими;
* таблиця попереднього перегляду має залишатися добре читабельною;
* статуси, помилки та попередження мають бути чітко помітними;
* кнопка зупинки завдання має бути достатньо помітною, але не агресивною в неактивному стані.

Приклад логіки налаштування теми:

```json
{
  "interface": {
    "theme": {
      "mode": "light | dark | system",
      "default": "light",
      "darkTheme": {
        "usePureBlack": false,
        "background": "soft-dark",
        "surface": "graphite",
        "contrast": "comfortable"
      }
    }
  }
}
```

Вибрана тема має зберігатися між запусками програми.

Перемикання теми має застосовуватися до всього інтерфейсу:

* головного вікна;
* правого сайдбару;
* вкладок;
* таблиці попереднього перегляду;
* діалогових вікон;
* попереджень;
* прогрес-бару;
* кнопок;
* полів введення;
* списків;
* комбобоксів;
* елементів вибору моделей;
* елементів керування API-профілями.

## 6. Правий сайдбар

У правому сайдбарі потрібно реалізувати три вкладки:

1. Папки.
2. Архіви.
3. Налаштування.

### 6.1. Вкладка “Папки”

У цій вкладці користувач має вибирати папки для обробки.

Потрібно передбачити:

* додавання однієї або кількох папок;
* видалення папки зі списку;
* очищення списку;
* відображення повного шляху;
* можливість увімкнути або вимкнути конкретну папку перед аналізом;
* індикацію недоступних або неіснуючих папок;
* збереження останніх вибраних папок, якщо це відповідає логіці програми.

### 6.2. Вкладка “Архіви”

У цій вкладці користувач має вибирати архіви для обробки.

Потрібно передбачити:

* додавання одного або кількох архівів;
* видалення архіву зі списку;
* очищення списку;
* відображення повного шляху до архіву;
* можливість увімкнути або вимкнути конкретний архів перед аналізом;
* індикацію недоступних або пошкоджених архівів, якщо це можливо визначити на етапі сканування.

### 6.3. Вкладка “Налаштування”

У цій вкладці потрібно розмістити налаштування аналізу, моделей, API, теми та мови.

Налаштування мають бути згруповані за логічними блоками:

1. Інтерфейс.
2. Загальні обмеження довжини.
3. Звичайний аналіз.
4. Інтелектуальний аналіз.
5. Провайдер інтелектуального аналізу.
6. Ollama.
7. Claude.
8. OpenAI.
9. Grok.
10. Gemini.
11. Словник скорочень.
12. Кешування.
13. Пакетна обробка.
14. Безпека збереження ключів.

Програма має коректно обробляти зупинку під час:

* сканування;

* інтелектуального аналізу;

Приклади статусів:

* переміщення файлів;
* обробка архівів;
* завершено;
* зупинено користувачем;
* помилка.

## 8. Звичайний аналіз

Кнопку “Розрахувати” потрібно замінити на “Проаналізувати”.

Звичайний аналіз має працювати без ШІ-моделі. Він має використовувати алгоритмічні правила скорочення та словник.

У вкладці “Налаштування” потрібно додати блок для звичайного аналізу.

У цьому блоці мають бути чекбокси та поля налаштувань.

### 8.1. Базові алгоритмічні скорочення

Потрібно реалізувати такі опції:

* скорочення дат;
* обрізання праворуч;
* обрізання кожного слова до заданої кількості символів. Якщо слова розділені не пробілами, а символом _ (наприкдад 2._Продовження_строку_відрядження_судді_(тимчасове_переведення)), то скорочення кожного слова, розділеного _;
* видалення зайвих пробілів;
* заміна кількох розділювачів одним;
* видалення або нормалізація службових символів;
* скорочення типових слів за словником;
* видалення повторюваних частин шляху, якщо це безпечно;
* нормалізація регістру;
* транслітерація, якщо це потрібно для файлової системи;
* видалення технічних маркерів, якщо вони не несуть змісту;
* скорочення службових юридичних слів за словником;
* видалення дубльованих слів;
* нормалізація дат до коротшого формату;
* видалення надмірних пробілів навколо дефісів, підкреслень і дужок;
* заміна довгих типових фраз на затверджені короткі форми.

### 8.2. Умова застосування алгоритмічних скорочень

Алгоритмічне скорочення має застосовуватися лише тоді, коли довжина перевищує допустимі межі.

Наприклад, якщо ввімкнено “обрізання кожного слова до 5 символів”, це правило не повинно застосовуватися до назв, які вже вкладаються у встановлений ліміт.

Правило “обрізання кожного слова до заданої кількості символів” має бути крайнім варіантом, а не стандартною поведінкою.

### 8.3. Результат звичайного аналізу

За результатами звичайного аналізу система має запропонувати нові назви та нові шляхи.

Користувач має бачити:

* початковий шлях;
* запропонований новий шлях;
* які правила були застосовані;
* чи перевищує шлях допустиму довжину після аналізу;
* чи є попередження;
* чи потрібне ручне втручання;
* джерело кожної зміни: словник, алгоритм, ручне редагування або ШІ.

## 9. Інтелектуальний аналіз

Поруч із кнопкою “Проаналізувати” потрібно додати кнопку “Інтелектуальний аналіз”.

Інтелектуальний аналіз — це аналіз за допомогою ШІ-моделі.

Під час такого аналізу система має передавати моделі лише ті назви або частини шляхів, які справді потребують скорочення, і отримувати структуровану відповідь із варіантами скорочень.

### 9.1. Коли запускати ШІ-аналіз

ШІ-аналіз має запускатися лише для елементів, які:

* перевищують допустиму довжину;
* не можуть бути достатньо скорочені звичайними алгоритмами;
* потребують контекстного скорочення;
* не мають готового скорочення у словнику;
* не були знайдені в локальному кеші результатів.

Не можна передавати до моделі всі шляхи без фільтрації.

### 9.2. Оптимізація передавання шляхів

Не потрібно передавати кожний шлях повністю, якщо значна частина шляхів повторюється.

Потрібно групувати шляхи за спільними частинами.

Наприклад, якщо багато файлів лежать в одній папці, спільну частину шляху потрібно передати один раз, а у структурі запиту використовувати посилання на цю частину.

Мета:

* зменшити кількість токенів;
* зменшити кількість запитів;
* пришвидшити обробку;
* зменшити витрати при використанні платних API;
* зменшити навантаження на локальну Ollama;
* уникнути повторної обробки однакових фрагментів.

### 9.3. Формат відповіді від моделі

Модель має повертати результат у структурованому форматі, який можна автоматично обробити у програмі.

Бажаний формат:

```json
{
  "items": [
    {
      "id": "segment_001",
      "original": "Конституційний Суд України",
      "suggested": "КСУ",
      "type": "known_abbreviation",
      "confidence": 0.98,
      "reason": "Стала загальноприйнята абревіатура"
    }
  ],
  "warnings": []
}
```

Для неприйнятних скорочень модель має повертати оригінал або безпечний варіант без штучної абревіатури.

Приклад:

```json
{
  "items": [
    {
      "id": "segment_002",
      "original": "_ЛЕВЧУК_О_О_Кандзюба",
      "suggested": "_ЛЕВЧУК_О_О_Кандзюба",
      "type": "no_change",
      "confidence": 1,
      "reason": "Назва містить прізвища та ініціали. Створення штучної абревіатури заборонене."
    }
  ],
  "warnings": []
}
```

### 9.4. Валідація відповіді моделі

Після отримання відповіді від моделі потрібно виконати локальну валідацію.

Програма не повинна сліпо застосовувати відповідь моделі.

Потрібно перевірити:

* чи відповідь є валідним JSON;
* чи всі елементи мають очікувані поля;
* чи не містить скорочення заборонених штучних абревіатур;
* чи не збільшує скорочення довжину;
* чи не створює конфлікт назв;
* чи не порушує правила щодо прізвищ, імен та ініціалів;
* чи не пропонує модель змінити назву, яка не потребує скорочення.

Якщо відповідь моделі некоректна, елемент потрібно позначити як такий, що потребує ручного втручання.

## 22. Словник скорочень

Потрібно підтримувати словник скорочень.

Кожен запис має містити:

```json
{
  "key": "конституційни суд україн",
  "original": "Конституційний Суд України",
  "short": "КСУ",
  "source": "dict",
  "approved": true
}
```

Джерела можуть бути такими:

* `dict` — скорочення із затвердженого словника;
* `algorithm` — скорочення створене звичайним алгоритмом;
* `llm` — скорочення запропоноване моделлю;
* `manual` — скорочення введене користувачем.

Скорочення з джерелом `llm` не повинні автоматично потрапляти до постійного словника без підтвердження користувача.

Потрібно передбачити можливість:

* переглядати словник;
* додавати нові скорочення;
* редагувати скорочення;
* видаляти скорочення;
* підтверджувати LLM-скорочення перед додаванням до словника;
* відхиляти LLM-скорочення;
* позначати скорочення як затверджені.

## 23. Валідація скорочень

Перед застосуванням скорочення потрібно виконати перевірку.

Скорочення не можна застосовувати, якщо:

* воно порожнє;
* воно складається з незрозумілого набору літер;
* воно створене з прізвищ або ініціалів;
* воно суттєво спотворює зміст;
* воно збігається з іншим скороченням, але має інше значення;
* воно збільшує ризик конфлікту назв;
* воно не зменшує довжину;
* воно не потрібне через відсутність перевищення ліміту;
* воно містить недопустимі символи файлової системи;
* воно робить назву занадто нечіткою для користувача.

## 24. Конфлікти назв

Після формування нових шляхів потрібно перевіряти конфлікти.

Потрібно виявляти:

* однакові нові назви файлів в одній папці;
* однакові нові назви папок;
* конфлікт із уже наявним файлом;
* конфлікт із уже наявною папкою;
* недопустимі символи;
* перевищення максимальної довжини після скорочення;
* порожні назви;
* назви, які складаються лише з розділювачів або службових символів;
* зарезервовані системні назви, якщо це актуально для файлової системи.

Конфлікти мають відображатися в таблиці попереднього перегляду.

До усунення критичних конфліктів перейменування або переміщення не повинно запускатися.

## 25. Виконання змін

Перейменування, переміщення та обробка архівів мають запускатися лише після попереднього перегляду.

Користувач має бачити фінальний список змін перед застосуванням.

Потрібно передбачити:

* підтвердження перед виконанням;
* журнал виконання;
* обробку помилок;
* можливість зупинити операцію;
* позначення успішно оброблених елементів;
* позначення елементів із помилками;
* неможливість запуску при критичних конфліктах;
* повторну перевірку перед фактичним застосуванням змін.

## 26. Вимоги до промпту для ШІ-моделі

Промпт має чітко забороняти моделі створювати довільні абревіатури.

Модель повинна:

* скорочувати лише те, що справді потребує скорочення;
* не змінювати назви, які вкладаються в допустиму довжину;
* використовувати сталі абревіатури;
* не створювати абревіатури з прізвищ, імен та ініціалів;
* повертати структурований JSON;
* пояснювати причину кожної зміни;
* позначати випадки, коли зміну робити не потрібно;
* не вигадувати скорочення без підстав;
* не скорочувати довільні фрагменти до незрозумілих наборів літер;
* повертати оригінал, якщо безпечне скорочення неможливе.

### 26.1. Базова логіка промпту

Промпт має містити такі правила:

```text
Ти отримуєш фрагменти шляхів, назв файлів або назв папок.
Твоє завдання — запропонувати скорочення лише тоді, коли це справді потрібно.

Заборонено створювати штучні абревіатури з імен, прізвищ, ініціалів, назв сторін або учасників справи.
Якщо назву не потрібно скорочувати або безпечне скорочення неможливе, поверни оригінал без змін.

Використовуй тільки сталі, зрозумілі, загальноприйняті або словникові скорочення.
Відповідь поверни тільки у валідному JSON.
```

## 27. Безпека збереження налаштувань

API-ключі не можна зберігати у відкритому вигляді.

Потрібно передбачити захищене або зашифроване збереження для:

* Claude API-ключів;
* OpenAI API-ключів;
* Grok API-ключів;
* Gemini API-ключів;
* інших чутливих параметрів, якщо вони з’являться.

Налаштування кожного провайдера мають зберігатися незалежно.

Видалення профілю має вимагати підтвердження користувача.

## 28. Журналювання

Потрібно вести журнал виконання операцій.

У журналі потрібно фіксувати:

* запуск сканування;
* завершення сканування;
* запуск звичайного аналізу;
* запуск інтелектуального аналізу;
* вибраного провайдера;
* кількість елементів у пакетах;
* помилки API;
* помилки парсингу відповіді моделі;
* зупинку користувачем;
* перейменування файлів;
* перейменування папок;
* переміщення файлів;
* обробку архівів;
* конфлікти;
* пропущені елементи.

Журнал не повинен містити відкритих API-ключів.
